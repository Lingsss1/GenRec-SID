# 数据处理详解

## 一、Phase 1：数据载入与 5-core 过滤

### 1.1 数据源

- **来源**：HuggingFace `McAuley-Lab/Amazon-Reviews-2023`
- **类目**：Cell_Phones_and_Accessories
- **原始格式**：JSONL.GZ（`Cell_Phones_and_Accessories.jsonl.gz`）

### 1.2 字段提取

从原始数据中提取三个核心字段：
- `user_id`：用户标识
- `parent_asin`：商品标识（父 ASIN）
- `timestamp`：交互时间戳

### 1.3 5-core 过滤

采用迭代式 5-core 过滤，确保数据质量：

```
重复以下步骤（最多 20 轮，直到收敛）：
  1. 统计每个 user 的交互数，保留 ≥ 5 的用户
  2. 统计每个 item 的交互数，保留 ≥ 5 的商品
  3. 如果本轮无数据被过滤，则收敛退出
```

### 1.4 元数据加载

加载商品元数据，包含：
- `title`：商品标题
- `store`：店铺名称
- `categories`：商品分类

### 1.5 输出

| 文件 | 说明 |
|------|------|
| `data/raw/interactions_5core.parquet` | 5-core 过滤后的交互数据 |
| `data/raw/item_metadata.parquet` | 商品元数据 |
| `data/raw/data_statistics.json` | 数据统计信息 |

## 二、Phase 2：嵌套候选集构造

### 2.1 设计目标

构造 4 个**严格嵌套**的商品子集，用于研究候选集大小 N 对推荐性能的影响：

```
S1 (5K) ⊂ S2 (20K) ⊂ S2.5 (50K) ⊂ S3 (100K)
```

### 2.2 分层抽样方法

**步骤 1：Log-binning**
- 计算每个商品的 popularity（交互次数）
- 对 popularity 做对数变换后分为 **20 个桶**
- 目的：让抽样覆盖从冷门到热门的完整分布

**步骤 2：按比例分层抽样**
- 从大到小嵌套抽样：S3(100K) → S2.5(50K) → S2(20K) → S1(5K)
- 每一层从上一层中按桶内比例抽样
- 保证各子集的 popularity 分布与全集一致

### 2.3 嵌套验证

抽样完成后验证：
- S1 的所有商品都在 S2 中
- S2 的所有商品都在 S2.5 中
- S2.5 的所有商品都在 S3 中

### 2.4 输出

| 文件 | 说明 |
|------|------|
| `data/subsets/S1_items.txt` | 5K 商品 ID 列表 |
| `data/subsets/S2_items.txt` | 20K 商品 ID 列表 |
| `data/subsets/S2.5_items.txt` | 50K 商品 ID 列表 |
| `data/subsets/S3_items.txt` | 100K 商品 ID 列表 |
| `data/subsets/subset_statistics.json` | 各子集统计信息 |

## 三、Phase 2.5：数据质量验证

### 3.1 验证项目

| 检查项 | 标准 |
|--------|------|
| 元数据覆盖率 | ≥ 95% 的商品有 title |
| 时间戳格式 | Unix 毫秒，合理范围内 |
| 交互去重 | 同一 (user, item) 对只保留最早一条 |
| 序列长度 | 过滤长度 < 3 的用户 |

### 3.2 去重处理

- 输入：`interactions_5core.parquet`
- 去重规则：同一 (user_id, parent_asin) 保留最早的交互
- 输出：`interactions_5core_dedup.parquet`

### 3.3 Leave-one-out 切分验证

预演 Phase 3 的数据划分策略，确保逻辑正确。

## 四、Phase 4：SID 生成

### 4.1 语义嵌入

- **模型**：`Qwen/Qwen3-Embedding-0.6B`（输出维度 1024）
- **输入文本**：拼接 `title + store + main_category`
- **池化方式**：Last-token pooling（左填充 + attention mask）
- **策略**：仅在 S3（最大子集，100K 商品）上生成嵌入并训练 RQ-VAE，冻结后复用编码所有子集

### 4.2 FAISS Residual Quantizer（RQ-VAE）

| 参数 | 值 |
|------|-----|
| 输入维度 | 1024 |
| 层数（n_levels） | 3 |
| 每层 codebook 大小 | 256 |
| 总 SID 空间 | 256³ = 16,777,216 |
| 训练方式 | K-means 初始化 |
| 编码方式 | 贪心（max_beam_size=1） |
| S3 碰撞率 | 约 35.9% |

**RQ-VAE 原理**：
1. 第 1 层：在 codebook_0 中找到最近的码字，得到 `<a_X>`
2. 计算残差 = 原始向量 - 码字_0
3. 第 2 层：在 codebook_1 中对残差找最近码字，得到 `<b_Y>`
4. 计算新残差 = 残差 - 码字_1
5. 第 3 层：在 codebook_2 中对新残差找最近码字，得到 `<c_Z>`

### 4.3 SID 格式

每个商品被编码为 3 个 token：

```
<a_X><b_Y><c_Z>    其中 X, Y, Z ∈ [0, 255]
```

与 MiniOneRec 的 `generate_indices.py` 格式完全兼容。

### 4.4 Prefix Trie 构建

为每个子集构建扁平化前缀树，用于推理时的约束解码：

```json
{
  "()":                        ["<a_0>", "<a_1>", ...],       // 根 → 合法的第 1 层 token
  "('<a_0>',)":                ["<b_5>", "<b_12>", ...],      // 第 1 层 → 合法的第 2 层 token
  "('<a_0>', '<b_5>')":        ["<c_3>", "<c_99>", ...]       // 第 2 层 → 合法的第 3 层 token
}
```

### 4.5 输出

| 文件 | 说明 |
|------|------|
| `embeddings/rq_vae/faiss_rq.index` | 训练好的 FAISS RQ 索引 |
| `embeddings/rq_vae/config.json` | RQ-VAE 配置 |
| `embeddings/S3_item_embeddings.pt` | S3 商品嵌入向量 |
| `embeddings/sids/S*_sid_mapping.json` | 各子集 SID 映射（含 token_list 和 str_sid） |
| `trie/S*_prefix_trie.json` | 各子集前缀树 |
