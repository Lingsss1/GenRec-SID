## 版本一：完整版（适合详细项目经历，约 300 字）

### 生成式序列推荐系统的 Catalog-Size Scaling Law 研究

**技术栈**：Qwen2.5-0.5B / Qwen3-Embedding / FAISS RQ-VAE / Transformers / PyTorch / Beam Search

- 基于 MiniOneRec 框架，构建了完整的生成式序列推荐管线：利用 RQ-VAE 将商品文本嵌入离散化为 3 层语义 ID（SID），通过多任务 SFT 微调 Qwen2.5-0.5B，使 LLM 根据用户交互历史直接生成目标商品 SID，并设计 Prefix Trie 约束解码保证输出合法性
- 在 Amazon Reviews 2023（Cell Phones，100K+ 商品）上设计了嵌套候选集实验（5K⊂20K⊂50K⊂100K），采用 Log-binning 分层抽样保持 popularity 分布一致，Leave-Last-Out 划分 + C=10 限流构建训练/验证/测试集
- 完成 4 模型 × 3 Beam Size 共 12 组评估实验，首次量化了生成式推荐在候选集大小 N 维度上的 Scaling Law：HR@10(N) = αN^(-β) + γ，拟合 R²=0.995，发现衰减指数 β≈0.59（N 每增大 10 倍，HR 降至 0.26 倍）
- 通过 AIC/BIC 模型选择、参数置信区间、残差分析和跨 Beam 一致性检验（β 标准差 0.047）验证了幂律关系的稳健性，并明确指出 DOF=1 的统计局限性

## 版本四：英文版（适合英文简历）

### Generative Sequential Recommendation: Catalog-Size Scaling Law

**Tech Stack**: Qwen2.5-0.5B, RQ-VAE, FAISS, Transformers, Constrained Beam Search

- Built an end-to-end generative recommendation pipeline: RQ-VAE semantic ID encoding → multi-task SFT on Qwen2.5-0.5B → Prefix Trie constrained decoding, evaluated on Amazon Reviews 2023 (100K+ items)
- Designed nested catalog experiments (5K⊂20K⊂50K⊂100K) with stratified sampling to control popularity distribution; conducted 12 evaluation runs (4 models × 3 beam sizes)
- First to quantify the catalog-size scaling law for generative recommendation: HR@10(N) = αN^(-β) + γ with β≈0.59 and R²=0.995; HR@10 drops 40% as N scales from 5K to 100K
- Validated robustness via AIC/BIC model selection, cross-beam consistency (β std=0.047), and residual analysis; identified DOF=1 limitation and proposed validation experiments

## 面试常见问题准备

### Q1：这个项目解决什么问题？

传统推荐系统依赖"召回+排序"两阶段架构，而生成式推荐让 LLM 直接生成目标商品 ID，实现端到端推荐。本项目研究的核心问题是：当候选商品集从 5K 扩展到 100K 时，这种生成式方法的性能如何衰减？衰减是否遵循可预测的规律？

### Q2：为什么用 RQ-VAE 而不是直接用商品 ID？

商品原始 ID（如 ASIN）是无语义的字符串，LLM 无法理解。RQ-VAE 将商品的文本嵌入（由 Qwen3-Embedding 生成）通过残差量化离散化为 3 层 token（如 `<a_34><b_215><c_140>`），使得语义相近的商品拥有相似的 SID 前缀，便于 LLM 学习和生成。

### Q3：Scaling Law 的核心发现是什么？

HR@10 随候选集大小 N 遵循幂律+饱和衰减：HR@10(N) = 3.30 × N^(-0.59) + 0.021，R²=0.995。衰减指数 β≈0.59 意味着 N 每增大 10 倍，HR 降至原来的 26%。这个衰减速度显著快于模型参数量维度（Kaplan β≈0.08）和数据量维度（DLRM β≈0.28），说明候选集扩展是生成式推荐最严峻的 scaling 挑战。

### Q4：R²=0.995 是否说明拟合很好？

需要谨慎解读。我们只有 4 个数据点拟合 3 个参数（DOF=1），R² 存在机械性膨胀。但以下证据支持结论的合理性：(1) 幂律形式有 Scaling Laws 文献的理论先验；(2) β 值在 3 个 Beam 档位下一致（std=0.047）；(3) 残差无系统偏差。建议补充 N=10K/30K/70K 实验提升统计功效。

### Q5：Prefix Trie 约束解码是怎么工作的？

在 Beam Search 的每一步，通过自定义的 ConstrainedLogitsProcessor 将所有不合法 token 的 logits 设为 -∞。合法 token 由 Prefix Trie 决定：第 1 步只允许生成存在的 `<a_*>`，第 2 步根据已生成的 `<a_X>` 只允许对应的 `<b_*>`，第 3 步类似。这保证了输出的 SID 一定是候选集中存在的商品。

### Q6：多任务 SFT 的三个任务分别是什么？为什么这样设计？

(1) 主任务：历史 SID 序列 → 目标 SID（核心推荐能力）；(2) SID ↔ 标题双向预测（建立 SID 与语义的对齐）；(3) 历史 SID → 目标标题（融合语义理解）。辅助任务让模型不仅学会 SID 之间的序列模式，还理解 SID 背后的语义信息，提升泛化能力。

### Q7：如果要提升大候选集下的性能，你会怎么做？

四个方向：(1) 增大模型规模（0.5B→3B/7B），预期 β 值降低；(2) 改进 SID 编码（增加 RQ 层数或 codebook 大小，降低碰撞率）；(3) 分层架构（召回截断到 20K + 生成式精排）；(4) 放宽 C=10 限流，增加训练样本多样性。
