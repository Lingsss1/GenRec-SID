# 评估与分析详解（Phase 6 & 7）

## 一、评估流程（Phase 6）

### 1.1 约束解码

生成式推荐的核心挑战是：模型可能生成不存在的 SID。本项目通过 **Prefix Trie 约束解码** 解决这一问题。

#### ConstrainedLogitsProcessor

在 Beam Search 的每一步，通过自定义的 `ConstrainedLogitsProcessor` 限制模型只能生成合法的下一个 token：

```
推理过程（生成 <a_X><b_Y><c_Z>）：

Step 1: 根据 hash_dict["()"]，只允许生成合法的 <a_*> token
Step 2: 根据 hash_dict["(<a_X>,)"]，只允许生成合法的 <b_*> token
Step 3: 根据 hash_dict["(<a_X>, <b_Y>)"]，只允许生成合法的 <c_*> token
```

**实现方式**：
- 将所有不合法 token 的 logits 设为 `-inf`
- 合法 token 的 logits 保持不变
- 保证 Beam Search 只探索合法的 SID 空间

#### hash_dict 构建

从 `index.json` 构建 `hash_dict`：
1. 将每个 SID 拼成 `"### Response:\n<a_X><b_Y><c_Z>\n"`
2. 用 tokenizer 编码为 token ID 序列
3. 从 `prefix_index` 开始，构建前缀 → 合法下一个 token 的映射

> 注：`hash_dict` 与 `SIDTrie` 在功能上等价。当前评估代码使用 `hash_dict`，`trie_file` 参数保留兼容性。

### 1.2 评估指标

#### HR@K（Hit Rate at K）

```
HR@K = 1  如果 ground_truth ∈ top-K 预测列表
HR@K = 0  否则
```

#### NDCG@K（Normalized Discounted Cumulative Gain at K）

```
NDCG@K = 1.0 / log₂(rank + 1)  如果命中，rank 为 ground_truth 在 top-K 中的位置
NDCG@K = 0                       如果未命中
```

默认 K = 10。

## 三、核心研究问题与预期发现

### Q1：HR@10 随 N 如何衰减？

**预期**：随着候选集从 5K 增大到 100K，HR@10 显著下降。原因：
- SID 空间变大，生成正确 SID 的难度增加
- RQ-VAE 碰撞率升高（S3 约 35.9%）
- 模型需要区分更多相似商品

### Q2：Beam Size 能否缓解衰减？

**预期**：增大 Beam Size 有一定收益，但边际递减。
- Beam=20 → 50 收益较大
- Beam=50 → 100 收益递减
- 大 N 下 Beam 收益更明显

### Q3：生成式推荐的实用边界在哪？

**预期**：
- N ≤ 20K 时，HR@10 较高，生成式推荐可用
- N = 100K 时，HR@10 可能低于业务阈值
- 建议大候选集场景下配合传统召回做截断
