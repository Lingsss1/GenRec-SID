# 测试集构建详解（Phase 3）

## 一、总体流程

对 S1 / S2 / S2.5 / S3 四个子集分别独立构建序列数据，每个子集产出 `train.csv`、`valid.csv`、`test.csv`。

```
原始交互 → 子集过滤 → 按用户分组排序 → 截断 → Leave-Last-Out 划分
                                                    ↓
                                         Train: 滑动窗口 + C=10 限流
                                         Valid: 倒数第 2 条
                                         Test:  最后 1 条
```

---

## 二、详细步骤

### 2.1 子集过滤

只保留 `parent_asin ∈ 当前子集` 的交互记录。例如处理 S2 时，只保留 S2 中 20K 商品的交互。

### 2.2 按用户分组排序

- 按 `user_id` 分组
- 每组内按 `timestamp` 升序排列，形成用户的时间序列

### 2.3 序列截断

- 每用户保留最近 **20 条**交互（`max_length=20`）
- 过滤掉长度 < 3 的序列（`min_length=3`），因为至少需要 1 条历史 + 1 条验证 + 1 条测试

### 2.4 Leave-Last-Out 划分

对于一个长度为 n 的用户序列 `[i₁, i₂, ..., iₙ]`：

| 数据集 | 输入（history） | 目标（target） |
|--------|----------------|---------------|
| **Test** | `[i₁, i₂, ..., iₙ₋₁]`（前 n-1 条） | `iₙ`（最后 1 条） |
| **Valid** | `[i₁, i₂, ..., iₙ₋₂]`（前 n-2 条） | `iₙ₋₁`（倒数第 2 条） |
| **Train** | 滑动窗口生成（见下） | — |

### 2.5 滑动窗口训练样本

对前 n-2 条交互做滑动窗口，生成多条训练样本：

```
用户序列: [A, B, C, D, E, F, G]  (n=7)

Test:  history=[A,B,C,D,E,F], target=G
Valid: history=[A,B,C,D,E],   target=F

Train 滑动窗口（对 [A,B,C,D,E,F] 做滑窗）:
  history=[A],         target=B
  history=[A,B],       target=C
  history=[A,B,C],     target=D
  history=[A,B,C,D],   target=E
  history=[A,B,C,D,E], target=F
```

### 2.6 C=10 限流

**问题**：热门商品作为 target 出现次数远多于冷门商品，导致训练集分布偏斜。

**解决方案**：每个 target item 最多保留 **10 条**训练样本（C=10）。

**优先策略**：优先保留来自不同用户的样本，增加多样性。

## 三、输出格式

### CSV 列结构

| 列名 | 说明 |
|------|------|
| `user_id` | 用户 ID |
| `history` | 历史交互商品序列（空格分隔的 parent_asin） |
| `target` | 目标商品 parent_asin |
| `context_length` / `seq_length` | 历史序列长度 |

### 后续处理（与 SID 映射合并）

Phase 4 生成 SID 后，CSV 文件会增加以下列：

| 列名 | 说明 |
|------|------|
| `history_item_sid` | 历史交互的 SID 列表（如 `['<a_34><b_215><c_140>', ...]`） |
| `item_sid` | 目标商品的 SID（如 `<a_11><b_244><c_31>`） |
| `history_item_title` | 历史交互的商品标题列表 |
| `item_title` | 目标商品标题 |

## 五、设计考量

### 5.1 为什么用 Leave-Last-Out 而非随机划分？

序列推荐任务中，时间顺序至关重要。Leave-Last-Out 保证：
- 测试集中的 target 一定是用户最后一次交互
- 不存在"未来信息泄露"问题
- 与学术界标准评估协议一致

### 5.2 为什么限制 C=10？

- 防止热门商品在训练集中过度表示
- 提升模型对长尾商品的推荐能力
- 控制训练集大小，避免计算资源浪费

### 5.3 为什么截断到 20 条？

- LLM 输入长度有限（`cutoff_len=512`）
- 过长序列中早期交互对当前推荐的贡献递减
- 20 条在 Cell Phones 类目下已覆盖大部分用户的有效交互历史
