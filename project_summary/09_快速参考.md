### 实验结果（HR@10）

| N | Beam=20 | Beam=50 | Beam=100 |
|---|---------|---------|----------|
| **5K** | 0.0424 | **0.0441** | 0.0450 |
| **20K** | 0.0308 | 0.0318 | 0.0324 |
| **50K** | 0.0256 | 0.0260 | 0.0268 |
| **100K** | 0.0250 | 0.0266 | 0.0264 |

### 幂律拟合公式（Beam=20）

```
HR@10(N) = 3.30 × N^(-0.59) + 0.021
R² = 0.995
```

### 关键指标

- **最高 HR@10**：0.0450（N=5K, Beam=100）
- **最大衰减**：5K→20K 下降 27.9%
- **总衰减**：5K→100K 下降 39.7%
- **Beam 收益**：20→50 约 +4%，50→100 约 +2%

## 六、常见问题

### Q1：为什么 N=100K 的 HR@10 不比 N=50K 低？

**A**：可能原因包括：
1. 模型容量瓶颈（0.5B 参数在 50K 附近达到上限）
2. RQ-VAE 碰撞饱和（实际可区分商品数约 64K）
3. C=10 限流导致 100K 中很多商品训练不充分
4. 测试集覆盖差异（50K 测试集可能包含更多训练充分的商品）

### Q2：为什么选择 Beam=50 而不是 Beam=100？

**A**：
- Beam=50 → 100 的收益 < 2%
- 但推理成本翻倍（时间和显存）
- 性价比最优点是 Beam=50

### Q3：如何提升大 N 下的性能？

**A**：四种策略：
1. 增大模型规模（1B / 3B / 7B）
2. 改进 SID 编码（增加 RQ 层数或 codebook 大小）
3. 分层架构（召回 + 生成式精排）
4. 放宽 C 限流（增加训练样本）

### Q4：生成式推荐适合哪些场景？

**A**：
- ✅ 垂直类目（N ≤ 20K）
- ✅ 个性化小候选集
- ✅ 需要语义理解的推荐
- ❌ 全站推荐（N > 50K）
- ❌ 对延迟要求极高的场景
