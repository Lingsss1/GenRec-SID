## 二、核心发现

### 发现 0：GRPO 强化学习有效提升推荐性能

在 M_N_5K 模型上，仅用 1000 条训练样本进行 1 epoch 的 GRPO 训练，即可获得一致性提升：

| 指标 | Beam=20 | Beam=50 | Beam=100 |
|------|---------|---------|----------|
| HR@10 提升 | +3.3% | +8.1% | +2.9% |
| NDCG@10 提升 | +4.5% | +3.2% | +0.0% |

**结论**：
- GRPO RL 在所有 beam size 下均提升或持平 HR@10，**最高提升 8.1%**（Beam=50）
- NDCG@10 在小 beam size 下提升更明显（+4.5%），说明 RL 不仅提高了命中率，还改善了排序质量
- Beam=100 时 NDCG 持平，可能因为 RL 训练数据量有限（仅 1000 样本），大 beam 下的排序优化空间未被充分利用
- 仅 1 epoch + 1000 样本即有效果，说明 GRPO 对生成式推荐有较好的适配性

**RL 训练设置**：
- 算法：GRPO（Group Relative Policy Optimization）
- 奖励函数：rule reward（完全匹配 = 1.0，否则 = 0.0）
- KL 惩罚系数 beta = 0.04
- 每个 prompt 生成 4 个候选（num_generations = 4）
- 硬件：2 x RTX 3090（主模型 + ref_model 分 GPU）

---

### 发现 1：HR@10 随 N 显著衰减

以 Beam=50 为基准：

| N | HR@10 | 相对于 N=5K 的衰减 |
|---|-------|-------------------|
| 5K | 0.0441 | 基准（100%） |
| 20K | 0.0318 | ↓ 27.9% |
| 50K | 0.0260 | ↓ 41.0% |
| 100K | 0.0266 | ↓ 39.7% |

**结论**：
- 从 5K 到 20K，HR@10 下降约 **28%**
- 从 5K 到 50K，HR@10 下降约 **41%**
- 从 50K 到 100K，性能基本持平（甚至略有回升），可能达到瓶颈

**解释**：
- 候选集增大 → SID 空间增大 → 生成正确 SID 的难度增加
- RQ-VAE 碰撞率升高（S3 约 35.9%），多个商品共享相同 SID
- 模型容量有限（0.5B 参数），难以记忆 100K 商品的 SID 映射

---

### 发现 2：Beam Size 收益有限且递减

#### N=5K 时的 Beam 收益

| Beam | HR@10 | 相对于 Beam=20 的提升 |
|------|-------|----------------------|
| 20 | 0.0424 | 基准 |
| 50 | 0.0441 | +4.0% |
| 100 | 0.0450 | +6.1% |

#### N=100K 时的 Beam 收益

| Beam | HR@10 | 相对于 Beam=20 的提升 |
|------|-------|----------------------|
| 20 | 0.0250 | 基准 |
| 50 | 0.0266 | +6.4% |
| 100 | 0.0264 | +5.6% |

**结论**：
- Beam 从 20 增大到 50，收益约 **4-6%**
- Beam 从 50 增大到 100，收益**几乎为 0**（甚至略有下降）
- 大 N 下 Beam 收益略大，但整体有限

**建议**：Beam=50 是性价比最优选择，Beam=100 计算成本翻倍但收益极小。

---

### 发现 3：N=50K 与 N=100K 性能接近

| Beam | N=50K | N=100K | 差异 |
|------|-------|--------|------|
| 20 | 0.0256 | 0.0250 | -2.3% |
| 50 | 0.0260 | 0.0266 | +2.3% |
| 100 | 0.0268 | 0.0264 | -1.5% |

**结论**：
- N=50K 和 N=100K 的 HR@10 几乎相同（差异 < 3%）
- 可能原因：
  1. 模型容量瓶颈：0.5B 参数难以有效利用 100K 候选集
  2. RQ-VAE 碰撞：S3 碰撞率 35.9%，实际可区分商品数约 64K
  3. 训练数据限流：C=10 限制导致 100K 商品中很多未充分训练

## 三、业务结论

### 3.1 生成式推荐的适用场景

| 候选集大小 | HR@10 (Beam=50) | 业务可用性 | 建议 |
|-----------|-----------------|-----------|------|
| **N ≤ 5K** | **0.0441** | ✅ 可用 | 直接使用生成式推荐 |
| **N = 20K** | 0.0318 | ⚠️ 边缘 | 可用，但性能下降明显 |
| **N ≥ 50K** | 0.0260 | ❌ 不推荐 | 配合召回截断到 20K 以内 |

### 3.2 推荐架构建议

**小候选集场景（N ≤ 20K）**：
```
用户历史 → 生成式推荐模型 → Top-10 商品
```

**大候选集场景（N > 50K）**：
```
用户历史 → 召回模型（截断到 20K） → 生成式推荐模型 → Top-10 商品
```

### 3.3 Beam Size 选择

- **推荐 Beam=50**：性价比最优
- Beam=20 性能略低（-4%），但推理速度快 2.5 倍
- Beam=100 性能提升极小（< 1%），但推理成本翻倍

## 四、与传统推荐方法对比

### 生成式推荐的优势

1. **端到端**：无需显式召回 + 排序两阶段
2. **语义理解**：利用 LLM 的语言能力理解商品语义
3. **灵活性**：可融合多种信号（序列、标题、类目等）
4. **可解释性**：通过辅助任务增强 SID 与语义的对齐

### 生成式推荐的局限

1. **候选集扩展性差**：N 增大时性能快速衰减
2. **计算成本高**：需要 LLM 推理，比传统向量检索慢
3. **SID 碰撞**：RQ-VAE 离散化导致信息损失
4. **模型容量依赖**：0.5B 模型难以记忆 100K 商品

## 五、未来改进方向

### 5.1 模型层面

- 增大模型规模（1B / 3B / 7B）
- 增加 RQ-VAE 层数（L=4 或 5）或 codebook 大小
- 改进 SID 编码方式（如 Hierarchical SID）

### 5.2 训练层面

- 放宽 C=10 限流，增加热门商品训练样本
- 增加训练 epoch 数
- ✅ **已验证**：引入 GRPO 强化学习，在 1000 样本 + 1 epoch 下即提升 HR@10 最高 8.1%
- 扩大 RL 训练数据量和 epoch 数，预期可获得更大提升
- 尝试其他奖励函数（ranking_reward、hierarchy_reward）以优化排序质量

### 5.3 推理层面

- 改进约束解码策略（如 Constrained Beam Search with Lookahead）
- 引入重排序（Reranking）模块
- 混合生成式 + 检索式（Hybrid Approach）

### 5.4 数据层面

- 扩展到更多类目（如 Books、Electronics）
- 引入多模态信号（商品图片、用户评论）
- 增强元数据（品牌、价格、评分等）

---

## 六、实验总结

本项目成功实现了一个完整的生成式序列推荐管线，从数据处理到模型训练再到评估分析，全流程自动化。

**主要贡献**：
1. 验证了生成式推荐在小候选集（N ≤ 20K）下的可行性
2. 量化了 N 增大对 HR@10 的衰减影响（5K → 100K 下降约 40%）
3. 发现 Beam Size 收益有限（Beam=50 最优）
4. 提出了大候选集场景下的混合架构建议
5. **验证了 GRPO 强化学习对生成式推荐的有效性**：仅 1000 样本 + 1 epoch 即提升 HR@10 最高 8.1%、NDCG@10 最高 4.5%

**局限性**：
- 模型规模较小（0.5B），未充分发挥生成式推荐潜力
- 单一类目数据，泛化性有待验证
- 未与传统推荐方法（如 GRU4Rec、SASRec）做对比实验
- RL 训练仅在 M_N_5K 上验证，未扩展到更大候选集
