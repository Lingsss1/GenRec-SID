## Q1: RQ-VAE 的 3 层 SID 是怎么设计的？

### 回答

**码本结构**：每层 codebook 大小 = 256（8 bits），3 层总 SID 空间 = 256³ = 16,777,216。每个商品被编码为 3 个 token：`<a_X><b_Y><c_Z>`，分别对应第 1/2/3 层量化码。

```
配置（phase4_sid_generation.py L41-50）：
  codebook_size: 256
  n_levels: 3
  sid_prefix: ['<a_{}>', '<b_{}>', '<c_{}>', '<d_{}>', '<e_{}>']
```

**残差量化的残差计算**：

RQ-VAE 的核心是逐层量化残差。设输入嵌入为 x，第 m 层的 codebook 为 T_m：

```
第 1 层：i_1 = argmin_j ||T_1[j] - x||²
         残差_1 = x - T_1[i_1]

第 2 层：i_2 = argmin_j ||T_2[j] - 残差_1||²
         残差_2 = 残差_1 - T_2[i_2]

第 3 层：i_3 = argmin_j ||T_3[j] - 残差_2||²

最终 SID = (i_1, i_2, i_3) → "<a_{i_1}><b_{i_2}><c_{i_3}>"
```

实现上使用 FAISS 的 `ResidualQuantizer`，采用贪心编码（`max_beam_size=1`）：

```python
# phase4_sid_generation.py L176-205
rq = faiss.ResidualQuantizer(d, num_levels, nbits)
rq.train_type = faiss.ResidualQuantizer.Train_default  # k-means 初始化
rq.max_beam_size = 1  # 贪心编码
rq.train(data_f32)
```

**为什么选 3 层而不是 2 层或 4 层**：

| 层数 | SID 空间 | 能否覆盖 100K 商品 | 生成步数 | 碰撞风险 |
|------|----------|-------------------|---------|---------|
| 2 层 | 256² = 65,536 | 不够（S3 有 100K 商品） | 2 步 | 极高 |
| **3 层** | **256³ = 16.7M** | **远超 100K，充分** | **3 步** | **S3 碰撞率 35.9%** |
| 4 层 | 256⁴ = 4.3B | 过剩 | 4 步 | 更低但增加生成成本 |

选择 3 层的核心权衡：
- 2 层空间不足以容纳 100K 商品（65K < 100K），碰撞率会极高
- 3 层空间 16.7M >> 100K，理论上碰撞率应该很低，但由于 RQ-VAE 的量化误差和嵌入分布的不均匀性，S3（100K）实际碰撞率仍达 35.9%
- 4 层会增加每次推理的生成步数（从 3 步到 4 步），约束解码的搜索空间也更大，但边际收益有限——碰撞主要来自嵌入空间的聚集性，增加层数不能根本解决

**是否做过消融**：没有。本项目固定 L=3、codebook=256，与 MiniOneRec 原始设置对齐。这是一个局限性，未来可以探索 L=4 或 codebook=512 的效果。代码已预留 `<d_{}>`、`<e_{}>` 前缀和任意层数的 trie 构建逻辑（`build_flat_trie` 支持任意 `n_levels`），扩展到 4-5 层只需修改配置。

## Q2: Prefix Trie 约束解码的具体实现是什么？

### 回答

**Trie 的构建对象**：对当前候选集中**所有合法 SID 序列**建立前缀索引。例如 S1 有 5000 个商品，就有 5000 条合法 SID 路径。

具体实现上不是传统的树结构，而是一个 **hash_dict**（扁平哈希表）：

```
key: 前缀 token ID 序列的字符串哈希（如 "123-456"）
value: 该前缀下所有合法的下一个 token ID 集合
```

构建过程（`grpo_trainer.py` `_build_hash_dict` L391-446）：
1. 从 `index.json` 读取所有商品的 SID token 列表
2. 拼接为 `"### Response:\n<a_X><b_Y><c_Z>\n"` 格式
3. 用 tokenizer 编码为 token ID 序列
4. 对每个前缀位置，记录 `prefix → {allowed_next_tokens}`

**约束方式：logits 层 mask，不是 beam search 剪枝**

```python
# grpo_trainer.py L106-131 (ConstrainedLogitsProcessor.__call__)
scores = torch.nn.functional.log_softmax(scores, dim=-1)
mask = torch.full_like(scores, float("-inf"))

for batch_id, beam_sent in enumerate(...):
    for beam_id, sent in enumerate(beam_sent):
        # 根据已生成的 token 查找合法的下一个 token
        prefix_allowed_tokens = self._prefix_allowed_tokens_fn(batch_id, hash_key)
        # 合法 token 位置设为 0（保留原始 logits），其余为 -inf
        mask[batch_id * self._num_beams + beam_id, prefix_allowed_tokens] = 0

scores = scores + mask  # 不合法 token 的 logits 变为 -inf
```

在 GRPO 训练中，还使用了 transformers 原生的 `prefix_allowed_tokens_fn` API（`make_prefix_allowed_fn` L457-484），效果等价但避免了 `LogitsProcessor` 的全局计数器问题。

**对推理延迟的影响**：

- 每个生成步需要一次 hash lookup（O(1)）+ 一次全 vocab 的 mask 操作
- 但 SID 只有 3 个 token，所以约束解码只在 3 步中生效
- 实际开销远小于模型前向传播本身
- 主要瓶颈在 beam search 的 beam_size × batch_size 组合，而非约束操作本身

## Q3: GRPO 里的"组内归一化 advantage"和标准 PPO 的 advantage 估计有什么本质区别？

### 回答

**本质区别：GRPO 不需要 value network（critic）**

| 维度 | GRPO | 标准 PPO |
|------|------|---------|
| Advantage 来源 | 组内相对奖励 | GAE（Generalized Advantage Estimation） |
| 是否需要 critic | 不需要 | 需要学习 V(s) |
| 基线（baseline） | 同组 reward 的均值 | 学习到的 V(s) |
| 归一化范围 | 组内（同一 prompt 的 G 个生成） | 通常是 batch-wise 或 global |
| 参数量 | 仅 policy model | policy + value model |

**GRPO 的 advantage 计算**：

```python
# grpo_trainer.py L704-710
# 将 rewards reshape 为 [num_prompts, num_generations]
mean_grouped_rewards = rewards.view(-1, self.num_generations).mean(dim=1)
std_grouped_rewards = rewards.view(-1, self.num_generations).std(dim=1)
# broadcast 回每个 completion
mean_grouped_rewards = mean_grouped_rewards.repeat_interleave(self.num_generations, dim=0)
std_grouped_rewards = std_grouped_rewards.repeat_interleave(self.num_generations, dim=0)
# 组内归一化
advantages = (rewards - mean_grouped_rewards) / (std_grouped_rewards + 1e-4)
```

公式：`A_i = (r_i - mean(r_group)) / (std(r_group) + 1e-4)`

直觉：在同一个 prompt 的 G 个生成中，reward 高于组均值的 completion 获得正 advantage（被鼓励），低于均值的获得负 advantage（被抑制）。

**Group 采样**：使用 `RepeatRandomSampler`（L137-158），每个 prompt index 重复 `num_generations` 次。本项目设 `num_generations=4`，即每个 prompt 生成 4 个候选 SID。

**KL 散度惩罚**：

使用 f-divergence 形式（而非标准 KL）：

```python
# grpo_trainer.py L757-762
per_token_kl = (
    torch.exp(ref_per_token_logps - per_token_logps)
    - (ref_per_token_logps - per_token_logps)
    - 1
)
```

即 `KL_t = exp(log π_ref - log π) - (log π_ref - log π) - 1`

这个形式的优势是非负且在 π = π_ref 时为 0，数值上比标准 KL 更稳定。

**beta 的调参**：beta=0.04 沿用 MiniOneRec 的默认值。beta 过大会让策略过于保守（不敢偏离 SFT 模型），beta 过小会导致策略崩溃（reward hacking）。0.04 在实验中表现稳定，RL 模型在 HR@10 上提升 3-8% 的同时没有出现 reward hacking。

**完整 GRPO loss**：

```python
# grpo_trainer.py L767-778
per_token_loss = torch.exp(per_token_logps - per_token_logps.detach()) * advantages.unsqueeze(1)
per_token_loss = -(per_token_loss - self.beta * per_token_kl)
loss = ((per_token_loss * completion_mask).sum(dim=1) / completion_mask.sum(dim=1)).mean()
```

## Q4: Log-binning 分层抽样保持 popularity 分布一致，popularity 是按什么定义的？

### 回答

**Popularity 定义：交互次数**（即 `parent_asin` 在交互数据中出现的频次）。

```python
# phase2_subset_construction.py L106-108
item_popularity = df['parent_asin'].value_counts()
all_items = item_popularity.index.tolist()
popularity_values = item_popularity.values
```

不是评分数量或评分均值，而是纯粹的交互频次。在 Amazon Reviews 2023 数据集中，一条交互 = 一条评论记录。

**Log-binning 分桶**：

```python
# phase2_subset_construction.py L35-57
bins = np.logspace(np.log10(min_pop), np.log10(max_pop), num_bins + 1)
bin_ids = np.digitize(popularity_values, bins) - 1
```

使用 20 个 log-scale 桶。log-scale 的原因是 popularity 分布本身是 long-tail / power-law 的：少数热门商品有数百次交互，大量长尾商品只有 5-10 次（5-core 过滤后最少 5 次）。线性分桶会导致绝大多数商品落入第一个桶。

**分层抽样**：

```python
# phase2_subset_construction.py L60-98
for bin_id in sorted(bin_counts.keys()):
    bin_ratio = len(items_in_bin) / total_items
    n_to_sample = max(1, int(target_size * bin_ratio))  # 至少抽 1 个
    sampled = np.random.choice(items_in_bin, size=n_to_sample, replace=False)
```

每个桶按其在总体中的比例抽样，保证子集的 popularity 分布形状与全集一致。

**长尾小桶的方差问题**：

这是一个合理的追问。确实存在以下问题：

1. **小桶样本量极少**：高 popularity 桶可能只有几个商品，`max(1, ...)` 保证至少抽 1 个，但 1 个样本的代表性很差
2. **抽样方差大**：假设某个桶有 3 个商品（popularity 分别为 500, 600, 700），抽 1 个的话，结果高度依赖随机种子
3. **嵌套抽样放大方差**：S3→S2.5→S2→S1 的嵌套结构意味着小桶中的商品在多次抽样中可能被反复选中或遗漏

**缓解措施**：
- `max(1, ...)` 保证每个桶至少有代表
- 嵌套抽样（S1 ⊂ S2 ⊂ S2.5 ⊂ S3）保证子集间的包含关系，避免不同子集间的分布漂移
- 固定 `seed=42` 保证可复现

**局限性**：没有做多次随机种子的稳定性分析。更严谨的做法是跑 5 个不同 seed 的子集，报告均值和标准差。

## Q5: Leave-Last-Out + C=10 限流是什么意思？

### 回答

**Leave-Last-Out 划分**：

对每个用户的交互序列（按时间排序），取最后一个交互作为 test，倒数第二个作为 valid，其余用于生成训练样本：

```
用户交互序列: [A, B, C, D, E]

Test:  history=[A,B,C,D], target=E
Valid: history=[A,B,C],   target=D
Train: 滑动窗口 →
  history=[A],       target=B
  history=[A,B],     target=C
  history=[A,B,C],   target=D  ← 包含 valid target，避免 context length gap
```

```python
# phase3_sequence_construction.py L59-105
# Valid: input=前 n-2, target=倒数第 2 个
# Test:  input=前 n-1, target=最后 1 个
# Train: 滑动窗口覆盖到 items[:-1]
```

**C=10 限流**：

C=10 是 **per-target-item** 的训练样本数上限，不是对用户序列的截断。

```python
# phase3_sequence_construction.py L107-178
# 对每个 target item：如果训练样本 > 10 条，只保留 10 条
# 优先选择不同 user 的样本（user 多样性优先）
```

**为什么需要 C=10**：
- 热门商品可能作为 target 出现在数千个训练样本中（因为很多用户都买过）
- 不限流会导致模型过度拟合热门商品，忽视长尾商品
- C=10 强制每个商品最多贡献 10 条训练样本，平衡了热门和长尾

**C=10 的依据**：
- 参考 MiniOneRec 原始设置
- 经验值：10 条样本足以让模型学到该商品的 SID 映射，同时不会主导训练
- 选择策略：优先选不同 user 的样本（增加多样性），次要按 context_length 排序（覆盖不同长度）

**用户历史只有 5 条交互怎么处理**：

```python
# phase3_sequence_construction.py L27, L47
# min_length=3：序列长度 < 3 的用户直接丢弃
```

- 历史 < 3 条：丢弃（无法同时产生 train/valid/test）
- 历史 = 3 条：valid 有 1 条历史，test 有 2 条历史，train 只有 1 条滑动窗口样本
- 历史 = 5 条：正常处理，train 产生 3 条滑动窗口样本
- C=10 不影响这些用户，因为它限制的是 target item 维度，不是 user 维度

另外，序列长度超过 20 条的用户会被截断到最近 20 条（`max_length=20`），保留最新的交互。

## Q6: Beam 50→100、N=100K 时 HR@10 下降 0.8%，这个负收益怎么解释？

### 回答

**数据**：N=100K 时，Beam=50 HR@10=0.0266，Beam=100 HR@10=0.0264，差异 = -0.0002（-0.8%）。

**核心判断：这个差异在统计上不显著。**

用两个比例的 z 检验：n=5000，p1=0.0266，p2=0.0264，z ≈ 0.09，远小于 1.96（95% 显著性阈值）。所以 0.8% 的"下降"完全可以用随机波动解释。

**但如果面试官追问"假设差异是真实的，可能的原因是什么"**：

1. **搜索空间过大导致低质量候选稀释**：Beam=100 会探索更多路径，但在 N=100K + 碰撞率 35.9% 的情况下，很多路径指向的是碰撞 SID（多个商品共享同一 SID）。更多 beam 可能找到更多碰撞 SID，但这些 SID 对应的商品不一定是 ground truth。

2. **模型置信度分布的长尾效应**：模型对 top-50 候选的 logits 差异可能已经很小，beam=100 额外找到的 50 个候选的 logits 更低，排序质量更差。在 HR@10 只看 top-10 的情况下，这些低质量候选可能"挤掉"原本在 top-10 的正确答案。

3. **约束解码的交互效应**：更大的 beam 意味着更多的并行路径需要约束，某些路径可能因为 trie 中的分支结构而被引导到次优方向。

4. **γ 饱和效应**：根据 Scaling Law 拟合，N=100K 时 HR@10 已接近渐近下限 γ≈0.021，此时任何超参数调整的边际效果都极小。

**结论**：最可能的解释是统计噪声。实际建议 Beam=50 作为性价比最优选择。

## Q7: 你用 z=0.09 说明 Beam 50 和 Beam 100 无显著差异，z 检验的假设是什么？

### 回答

**使用的检验**：两个独立比例的 z 检验（two-proportion z-test），pooled 版本。

**公式**：

```
p_pooled = (x1 + x2) / (n1 + n2)
z = (p1 - p2) / sqrt(p_pooled * (1 - p_pooled) * (1/n1 + 1/n2))
```

其中 p1=0.0266, p2=0.0264, n1=n2=5000。

**假设条件**：

1. **独立性**：两组样本独立。严格来说，Beam=50 和 Beam=100 在同一批测试样本上评估，样本不独立——这违反了独立性假设。
2. **大样本近似正态**：n*p > 5 且 n*(1-p) > 5。n=5000, p≈0.026, n*p=130 >> 5，满足。
3. **二值分布**：HR@10 对每个样本是 0 或 1（Bernoulli），满足。

**样本量是否足够**：

n=5000 对于检测 0.0002 的差异，统计功效（power）极低。要在 α=0.05 下以 80% power 检测 0.0002 的差异（从 0.0266 到 0.0264），需要的样本量约为：

```
n ≈ (z_α/2 + z_β)² * p*(1-p) * 2 / δ²
  ≈ (1.96 + 0.84)² * 0.0265 * 0.9735 * 2 / 0.0002²
  ≈ 约 400 万
```

所以 5000 样本完全不足以检测这个量级的差异，z=0.09 的结论（无显著差异）是合理的。

**更严谨的替代方案**：

1. **McNemar 检验**：因为两个 beam 在同一批样本上评估，样本是配对的。McNemar 检验利用配对信息，统计功效更高。
2. **Bootstrap 置信区间**：对 HR@10 的差值做 1000 次 bootstrap 重采样，直接估计差值的 95% CI。
3. **Permutation test**：不依赖分布假设的非参数检验。

**HR@10 用 z 检验是否合适**：合适。HR@10 本质上是 Bernoulli 比例（每个样本命中=1/未命中=0），两个比例的 z 检验正是为此设计的。唯一的问题是独立性假设被违反（同一批样本），所以 McNemar 检验更合适。

## Q8: 多任务 SFT 里"多任务"指的是哪几个任务？

### 回答

**三个任务**：

| 任务 | 数据集类 | 输入 → 输出 | 作用 |
|------|---------|------------|------|
| 主任务 | `SidSFTDataset` | 历史 SID 序列 → 目标 SID | 核心推荐能力 |
| 辅助任务 1 | `SidItemFeatDataset` | SID ↔ Title（双向） | SID-语义对齐 |
| 辅助任务 2 | `FusionSeqRecDataset` | 历史 SID 序列 → 目标 Title | 跨模态融合 |

**Prompt 示例**：

```
主任务：
  Input: History: <a_34><b_215><c_140>, <a_11><b_88><c_201>
  Response: <a_77><b_156><c_33>

辅助 1（SID→Title）：
  Input: SID: <a_77><b_156><c_33>
  Response: Apple iPhone 14 Pro Max Case

辅助 1（Title→SID）：
  Input: Title: Apple iPhone 14 Pro Max Case
  Response: <a_77><b_156><c_33>

辅助 2：
  Input: History: <a_34><b_215><c_140>, <a_11><b_88><c_201>
  Response: Samsung Galaxy S24 Screen Protector
```

**Loss 加权方式**：

没有显式的 per-task loss 权重。三个数据集通过 `ConcatDataset` 拼接后统一 shuffle 训练：

```python
# phase5_train.py L447-464
train_data = ConcatDataset([train_data1, train_data2, train_data3])
```

所有样本使用相同的 cross-entropy loss，隐式权重由各数据集的样本数量决定。例如在 M_N_5K 中，主任务约 2761 条，辅助任务 1 约 5000×2=10000 条（双向），辅助任务 2 约 2761 条，所以辅助任务 1 的隐式权重最大。

**Gradient conflict 问题**：

没有显式检测或处理 gradient conflict。这是一个局限性。

潜在的 conflict 场景：
- 主任务要求模型"记住" SID 之间的序列模式（偏记忆）
- 辅助任务 2 要求模型理解商品语义（偏理解）
- 两者的梯度方向可能不一致

可能的改进方案：
1. **PCGrad**（Project Conflicting Gradients）：检测梯度冲突时投影到非冲突方向
2. **GradNorm**：动态调整各任务的 loss 权重
3. **手动调权**：根据验证集表现调整各任务的采样比例
4. **Temperature sampling**：按 `T^(1/α)` 对各任务采样概率做温度缩放

实际上，从实验结果看，三任务联合训练的效果是正面的（MiniOneRec 论文中有消融实验证明辅助任务有帮助），说明在当前设置下 gradient conflict 不是主要问题。

## Q9: Qwen2.5-0.5B 相比更大的模型表现如何？

### 回答

**本项目只用了 Qwen2.5-0.5B-Instruct，没有做 scaling 实验。** 这是明确的局限性。

**但可以从 Scaling Law 分析中推断瓶颈所在**：

本项目拟合了 catalog-size 维度的 Scaling Law：

```
HR@10(N) = 3.30 × N^(-0.59) + 0.021, R² = 0.995
```

衰减指数 β ≈ 0.59，与其他 Scaling Law 对比：

| 研究 | 维度 | 衰减指数 β |
|------|------|-----------|
| **本项目** | **候选集大小 N** | **0.59** |
| Kaplan 2020 | 模型参数量 | 0.076 |
| DLRM 2022 | 训练数据量 | 0.28 |
| Chinchilla | 数据量 | 0.3-0.4 |

**关键洞察**：候选集大小维度的衰减指数（0.59）远大于模型参数量维度（0.076）。这意味着：

1. **主要瓶颈不在模型容量**：即使将模型从 0.5B 扩大到 7B（14 倍），按 Kaplan 的 β=0.076 估算，loss 改善约 `14^0.076 ≈ 22%`，对应的 HR@10 提升可能有限
2. **主要瓶颈在 SID 空间的组合爆炸**：N 从 5K 增大到 100K（20 倍），HR@10 下降 `20^0.59 ≈ 60%`，这个衰减速度远超模型增大带来的收益
3. **更大模型的预期收益**：在固定 N=5K 的情况下，更大模型可能通过更好的序列建模提升 HR@10；但在 N=100K 时，SID 碰撞率（35.9%）和 C=10 限流才是真正的瓶颈

**瓶颈分解**：

- **SID 生成的语言能力**：0.5B 模型生成 3 个 SID token 的能力应该足够（这不是复杂的语言生成任务）
- **用户交互历史的建模**：更大模型可能更好地捕捉长序列中的模式，但序列最长只有 20 条（`max_length=20`），0.5B 应该够用
- **SID 空间的记忆**：这才是真正的瓶颈。模型需要"记住" N 个商品的 SID 映射，0.5B 参数在 N≈50K 时可能达到记忆上限（N=50K 和 N=100K 的 HR@10 几乎相同，支持这个假设）

**未来实验建议**：在 N=5K 固定的情况下，对比 0.5B / 1.5B / 3B 的 HR@10，可以分离出"模型容量"维度的 scaling 行为。

## Q10: HR@10 作为唯一评估指标够吗？

### 回答

**本项目使用了两个指标：HR@10 和 NDCG@10**，不是只有 HR@10。

```python
# eval/run_eval.py L123-129
def compute_hr_ndcg(pred_list, ground_truth, k=10):
    pred_k = pred_list[:k]
    if ground_truth in pred_k:
        rank = pred_k.index(ground_truth) + 1
        ndcg = 1.0 / math.log2(rank + 1)
        return 1.0, ndcg
    return 0.0, 0.0
```

- **HR@10**：命中率，top-10 中是否包含 ground truth
- **NDCG@10**：排序质量，命中时考虑排名位置（排名越靠前分数越高）

**但确实缺少以下维度的评估**：

| 缺失指标 | 衡量什么 | 为什么重要 |
|---------|---------|-----------|
| Coverage@K | top-K 推荐覆盖了多少比例的商品 | 检测模型是否只推荐少数热门商品 |
| Diversity (ILS) | 推荐列表内部的多样性 | 避免推荐 10 个相似的手机壳 |
| Novelty | 推荐的商品有多"新颖" | 检测是否只推荐用户已知的热门商品 |
| Popularity Bias | 推荐结果的 popularity 分布 | 量化模型对热门商品的偏好程度 |

**Popularity bias 是否让 HR@10 虚高**：

这是一个很好的问题。存在以下风险：

1. **Cell Phones 类目特点**：热门商品（如 iPhone 手机壳、充电线）占据大量交互，模型可能学到"推荐热门商品总是对的"
2. **C=10 限流的缓解效果**：C=10 限制了每个热门商品的训练样本数，一定程度上缓解了 popularity bias，但不能完全消除——因为热门商品仍然更频繁地出现在用户历史中
3. **Leave-Last-Out 的影响**：test target 本身的 popularity 分布是自然的，热门商品更可能成为 test target，所以 HR@10 天然对能推荐热门商品的模型更友好

**如何验证是否虚高**：

1. **按 popularity 分组评估**：将 test 样本按 target item 的 popularity 分为热门/中等/长尾三组，分别计算 HR@10。如果模型只在热门组表现好，说明存在 bias
2. **计算 ARP（Average Recommendation Popularity）**：推荐列表中商品的平均 popularity，与 test set 的自然分布对比
3. **Calibration 分析**：推荐结果的 popularity 分布是否与用户实际交互的 popularity 分布匹配

**本项目的局限性**：确实没有做上述分析。但需要指出的是：
- 生成式推荐通过 SID 编码间接推荐商品，不像协同过滤那样直接利用 popularity 信号
- 约束解码保证推荐的都是合法商品，不存在"推荐不存在的商品"的问题
- GRPO RL 训练使用 rule reward（完全匹配），不直接鼓励推荐热门商品
